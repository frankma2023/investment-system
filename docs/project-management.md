## 一、项目概述

基于欧奈尔 CAN SLIM 投资理念，构建 A 股投资决策辅助系统。涵盖大盘市场扫描、行业相对强度分析、个股基本面 CAN SLIM 评分、形态与信号识别、回测验证、持仓和仓位管理六大核心功能，以独立看板页面展示。

## 二、技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Flask (Python 3) | 轻量 API 服务 |
| 前端架构 | 原生 HTML/CSS/JS | 静态页面 + ECharts |
| 图表库 | Apache ECharts 5.5 (CDN) | K线图/柱状图/折线图 |
| 数据库 | SQLite (WAL模式) | lixinger.db，2.8GB |
| 配置管理 | YAML + 自研简易解析器 | config/market/ 目录 |
| 测试框架 | pytest | — |
| 版本管理 | Git | GitHub 远程 |
| 数据源 | 理杏仁 API | 已有24个API脚本（src/data/lixr_api/） |
| 前端部署 | python3 -m http.server 8772 | 静态文件服务 |
| API 服务 | python3 src/server.py（端口8788） | Flask REST API |

## 五、已完成功能详情

### 5.1 抛盘日检测引擎 (`src/detectors/distribution_day.py`)

| 项目 | 说明 |
|------|------|
| 功能 | 识别4类抛盘日（标准/重抛盘/上影线/伪装抛盘日） |
| 配置 | `config/market/distribution_day.yaml`（4张卡片可调参数） |
| 看板 | `web/distribution-day/` — Xiaohongshu风格，参数面板 + K线图 + 明细表 |
| 特性 | 配置调整 → 保存到YAML → 追盘日看板联动读取 |

### 5.2 追盘日检测引擎 (`src/detectors/follow_through_day.py`)

| 项目 | 说明 |
|------|------|
| 功能 | 反弹尝试检测 + 追盘日确认 + 4类失效判定 |
| 配置 | `config/market/follow_through_day.yaml` |

**4类失效条件**（按优先级）：

| # | 条件 | 规则 | 参数 |
|---|------|------|------|
| 1 | 破位失效 | FTD后5日收盘 < FTD最低价 | `breakdown_days: 5` |
| 2 | 无力上涨 | FTD后10日最大涨幅 < 2% | `weak_days: 10, min_gain: 2.0` |
| 3 | 回撤过深 | FTD后15日回撤 > 涨幅的61.8%，且回撤≥2% | `retrace_days: 15, threshold: 0.618, min_retrace: 2.0` |
| 4 | 抛盘覆盖 | FTD后20日加权抛盘日 ≥ 3 | `cover_days: 20, count: 3` |

**回撤检测演进历史**：
- v1：阈值 50%，分母用 `ftd_close`（有bug）
- v2：阈值 50%，分母修正为 `peak_close`（"从最高点回撤"的标准定义）
- v3：阈值 61.8%（黄金分割），增加 2% 绝对回撤门槛，增加可关闭开关
- 验证：2020~2026 年 18 个失效 FTD 中，7 个因回撤失效。v3 规则下 1 个被救回（噪声过滤），2 个转由无力上涨接管

### 5.3 跨看板联动

| 特性 | 说明 |
|------|------|
| 导航链接 | 首页 / 抛盘日 / 追盘日 三向互链 |
| 配置持久化 | 面板调参 → 点💾保存 → 写入 YAML → 另一个看板回测读取最新参数 |
| 数据流 | `GET/POST /api/config` 双向同步面板与配置文件 |

### 5.4 回撤过深失效 — 历史回测分析

针对 2020~2026 年 18 个失效 FTD 的失效后 20 日走势分析：

| 指标 | 数值 |
|------|------|
| 失效后20T平均涨幅 | +1.28% |
| 失效后仍上涨比例 | 61%（11/18） |
| 强势恢复（>3%） | 33%（6/18） |
| 真正大跌（<−5%） | 17%（3/18） |

结论：失效条件整体偏保守（倾向于早判失效），但这符合体系设计目的——过滤假突破、避免基于不稳固信号重仓。2024-09-10 失效 → 2024-09-24 真 FTD 暴涨 28%，正是体系正常运作的案例。

## 七、开发方法

每个模块采用**四阶段迭代模型**：

```
产品说明书 → 回测原型 → 看板开发 → 回测验证
(周期1-2轮)   (周期1-2轮)  (周期1轮)    (周期1轮)
```

规则：
- ❌ 回测未通过，不得进入看板开发
- ✅ 每个阶段结束，用户确认后再进入下一阶段
- ✅ 每个原子任务完成后向用户同步进度
