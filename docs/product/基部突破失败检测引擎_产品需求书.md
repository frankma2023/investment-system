# 基部突破失败检测引擎 — 产品需求书 v1.0

> **核心原则**：不区分具体形态类型，对所有基部突破日统一监控。价格触发 + 成交量验证，三级严重级别区分。每次触发独立产出信号，支持逐日排查调参。

---

## 1. 产品概述

### 1.1 定位

本引擎是**基部突破检测引擎（Layer 1）的下游消费者**。它回答：

> 某个已确认的基部突破日，在后续 N 个交易日内是否出现了失败信号？失败的严重程度如何？

失败信号的判定基于两大维度：

1. **价格行为**：收盘价是否跌破关键价位（买点、突破日最低价、谷底支撑）
2. **成交量验证**：资金在回撤过程中的态度（缩量回踩 vs 放量溃败）

### 1.2 为什么是统一引擎而非按形态分拆

Layer 1 已经证明：8 种 O'Neil 基部形态共享同一个底层结构（上涨→前高→回调→谷→回升→突破），形态差异只影响标注，不影响"是否突破"的判断。

突破失败的检测面临完全同构的问题：所有形态的突破失败，本质上都是**「突破后 N 天内收盘价跌破关键价位」**。不同形态的关键价位（杯柄看柄部低点 vs 口袋支点看支点 K 线最低价）在 Layer 1 统一后均映射为通用字段：

| 统一字段 | Layer 1 输出 | 含义 |
|---|---|---|
| **买点** | `buy_point`（前高 + 0.01） | 枢轴点 |
| **突破日最低价** | K 线数据中的 `low` | 突破阳线的底线 |
| **谷底** | `trough_price` | 整个基部的最低支撑 |

因此无需为杯柄、碟形、双重底等分别实现失败检测，一个引擎监控所有 Layer 1 信号即可。

### 1.3 与 Layer 1 的关系

```
Layer 1: base_breakout.detect()
  └→ 输出: 基部突破信号（type: bullish）
        │
        └→ breakout_failure.detect() 消费 Layer 1 输出
              └→ 输出: 突破失败信号（type: bearish, severity: mild|confirmed|severe）
```

### 1.4 集成模式

引擎被 `engine_registry.py` 自动发现后：

- **pattern-scan 页面**：零改动自动显示卖出信号（买入 pin + 卖出五角星同屏）
- **独立回测看板**：`web/breakout-failure/` 提供参数调整 + 信号明细 + 单日排查

## 2. 核心概念

### 2.1 监控窗口

突破日之后 **N 个交易日**（默认 N=10）。窗口内每天收盘后对每个仍在窗口内的突破信号逐条检查触发规则。

### 2.2 触发规则（7 条）

| 规则 | 中文标签 | 条件 | 严重级别 | 触发时机 |
|---|---|---|---|---|
| **C** | 跌回枢轴点 | 收盘 < `buy_point` | **mild** | 当日收盘 |
| **G** | 抛盘日叠加 | 突破后 3 日内 ≥ 2 个抛盘日 + 任一交易日收盘 < `buy_point` | **confirmed** | 最早第2天收盘 |
| **H** | 窗口结束机构出货 | 第10天收盘：收盘 < `buy_point` + 窗口总成交量 > 同期均量 × 1.2 | **confirmed** | 第10天收盘 |
| **B** | 跌破突破日最低价 | 收盘 < 突破日 `low` | **severe** | 当日收盘 |
| **D** | 跌破谷底支撑 | 收盘 < `trough_price` × 0.98 | **severe** | 当日收盘 |
| **E** | 放量长阴砸穿 | 日跌幅 ≥ 3% + 量比 ≥ 1.5 + 收盘 < `buy_point` | **severe** | 当日收盘 |
| **F** | 连续放量溃败 | 连续 3 日收盘 < `buy_point` + 每日量 > 50 日均量 | **severe** | 第3天收盘 |

> 注：A~H 编号沿袭自 grilling 阶段。条件 A（缩量无效突破）不在失败引擎中处理，由 base_breakout 的 `breakout_vol_ratio` 参数自行控制。

### 2.3 严重级别

| 级别 | 含义 | 视觉 | 典型场景 |
|---|---|---|---|
| **mild** | 预警，可能只是正常回踩 | 橙色五角星 | 跌回买点但不放量 |
| **confirmed** | 确认失败，建议止损 | 紫色五角星 | 抛盘日叠加、机构出货 |
| **severe** | 形态彻底溃败，立即止损 | 黑色五角星 | 跌破最低价、放量长阴、连续溃败 |

### 2.4 信号产出策略

- **每次满足都触发**：同一突破在同一个监控窗口内，规则C可能触发多次（价格反复在买点上下穿越）
- **同天多条不合并**：如果第3天B、C、F同时满足，产出3条独立信号，每条带自己的规则代号和严重级别
- **不同突破独立监控**：同一只股票短期内可能有多笔突破，它们的监控窗口可以重叠，互不干扰

## 3. 量化计算规则

### 3.1 全局前置步骤

```
1. 调用 base_breakout.detect(klines, bp_params) 获取全部突破信号
2. 对每个突破信号 bs:
   2.1 从 klines 中定位 bs.signal_date 对应的 K 线，提取 breakout_low = kline.low
   2.2 确定监控窗口: monitor_start = bs.signal_date + 1 交易日, monitor_end = min(bs.signal_date + monitor_days, 数据末尾)
   2.3 遍历监控窗口内的每个交易日 d:
        逐一检查 7 条规则，触发则产出信号
```

### 3.2 单日基础数据

对每个监控日 `d`，从 K 线数据中提取：

| 变量 | 来源 | 说明 |
|---|---|---|
| `close` | `kline.close` | 当日收盘价 |
| `volume` | `kline.volume` | 当日成交量 |
| `vol_ratio` | `volume / sma(volume, 50)` | 当日量比（50 日均量不含当日） |
| `day_chg_pct` | `(close_t - close_{t-1}) / close_{t-1}` | 日间涨跌幅（方案 A） |
| `is_dd` | `volume > prev_volume && day_chg_pct < -0.2%` | 是否为抛盘日 |

### 3.3 规则 C：跌回枢轴点

```
条件: close < bs.buy_point
产出: { severity: 'mild', failed_rule: 'rule_c' }
每次满足都触发
```

### 3.4 规则 G：抛盘日叠加

```
窗口: [D+1, D+3]
计算窗口内抛盘日数量 dd_count
条件: dd_count ≥ 2 && 窗口内任一交易日 close < bs.buy_point
产出: { severity: 'confirmed', failed_rule: 'rule_g' }
每天收盘后重新计算，一旦满足立即产出，之后不再重复产出
```

### 3.5 规则 H：窗口结束机构出货

```
触发日: monitor_end（仅在第 monitor_days 天收盘后执行一次）
条件: close < bs.buy_point && 窗口总成交量 > 同期均量 × 1.2
  同期均量 = sum(volumes[monitor_start:monitor_end+1]) / monitor_days
产出: { severity: 'confirmed', failed_rule: 'rule_h' }
```

### 3.6 规则 B：跌破突破日最低价

```
条件: close < break_low（突破日 K 线的最低价）
产出: { severity: 'severe', failed_rule: 'rule_b' }
每次满足都触发
```

### 3.7 规则 D：跌破谷底支撑

```
条件: close < bs.trough_price × rule_d_trough_mult（默认 0.98）
产出: { severity: 'severe', failed_rule: 'rule_d' }
每次满足都触发
```

### 3.8 规则 E：放量长阴砸穿

```
条件: day_chg_pct ≤ -rule_e_chg_pct  (默认 -3.0%)
      && vol_ratio ≥ rule_e_vol_ratio  (默认 1.5)
      && close < bs.buy_point
产出: { severity: 'severe', failed_rule: 'rule_e' }
每次满足都触发
```

### 3.9 规则 F：连续放量溃败

```
条件: 连续 rule_f_days 个交易日（默认 3），每天同时满足:
        close < bs.buy_point
        volume > sma(volumes, vol_ma_days)
产出: { severity: 'severe', failed_rule: 'rule_f' }
仅在第3天收盘时触发一次（避免重复），后续交易日重新计数
```

## 4. 参数配置

### 4.1 YAML 配置文件

**路径**：`config/market/breakout_failure.yaml`

```yaml
breakout_failure:
  # 监控窗口
  monitor_days: 10                # 突破后监控交易日数（3~20）

  # 规则开关（可独立关闭某条规则）
  rule_b_enabled: true            # 跌破突破日最低价
  rule_c_enabled: true            # 跌回枢轴点
  rule_d_enabled: true            # 跌破谷底支撑
  rule_e_enabled: true            # 放量长阴砸穿
  rule_f_enabled: true            # 连续放量溃败
  rule_g_enabled: true            # 抛盘日叠加
  rule_h_enabled: true            # 窗口结束机构出货

  # 规则参数
  rule_d_trough_mult: 0.98        # 谷底倍数（0.90~0.99）
  rule_e_chg_pct: 3.0             # 长阴跌幅阈值 %（1~5）
  rule_e_vol_ratio: 1.5           # 长阴量比阈值（1.1~2.5）
  rule_f_days: 3                  # 连续放量天数（2~5）
  rule_f_vol_ratio: 1.0           # 每日量比阈值，>50日均量（0.8~1.5）
  rule_g_dd_threshold: -0.2       # 抛盘日跌幅阈值 %（-0.1~-1.0）
  rule_g_window: 3                # 抛盘日检测窗口交易日数（2~5）
  rule_g_count: 2                 # 抛盘日最少数量（1~4）
  rule_h_total_vol_ratio: 1.2     # 窗口总成交量/均量倍数（1.0~2.0）

  # 基础参数
  vol_ma_days: 50                 # 量能均线天数
```

### 4.2 参数说明

| 参数 | 类型 | 默认值 | 范围 | 用途 | 调参方向 |
|---|---|---|---|---|---|
| `monitor_days` | int | 10 | 3~20 | 监控窗口长度 | 缩短→更敏感（减少漏报）/ 延长→更保守 |
| `rule_*_enabled` | bool | true | — | 独立关闭某规则 | 回测时逐条评估各规则的贡献 |
| `rule_d_trough_mult` | float | 0.98 | 0.90~0.99 | 谷底支撑容忍度 | 降低→更难触发（更保守） |
| `rule_e_chg_pct` | float | 3.0 | 1~5 | 长阴定义 | 降低→更多信号（更敏感） |
| `rule_e_vol_ratio` | float | 1.5 | 1.1~2.5 | 放量定义 | 降低→更多信号 |
| `rule_f_days` | int | 3 | 2~5 | 连续下跌判定天数 | 增加→更难触发 |
| `rule_g_count` | int | 2 | 1~4 | 抛盘日最少数量 | 降低→更多信号 |
| `rule_h_total_vol_ratio` | float | 1.2 | 1.0~2.0 | 机构出货量能阈值 | 降低→更多信号 |

## 5. 信号输出格式

### 5.1 单条失败信号

```python
{
    # ── 标准字段（engine_registry 框架消费）──
    'date': '2026-05-18',           # 失败触发日
    'type': 'bearish',              # 卖出信号（固定值）

    # ── 关联字段 ──
    'breakout_date': '2026-05-15',  # 原始突破日（关联外键）
    'buy_point': 125.50,            # 原始突破买点
    'breakout_low': 124.00,         # 原始突破日最低价
    'failure_day': 3,               # 突破后第几个交易日

    # ── 判定字段 ──
    'failed_rule': 'rule_e',        # 触发规则代号
    'severity': 'severe',           # mild | confirmed | severe

    # ── 上下文信息（details 子对象）──
    'details': {
        'rule_label': '放量长阴砸穿',        # 中文标签
        'current_close': 123.50,             # 触发日收盘价
        'volume_ratio': 1.52,                # 触发日量比
        'day_chg_pct': -3.25,                # 触发日日间涨跌幅 %
        'trough_price': 98.00,               # 谷底价格（规则D需要）
        'breakout_vol_ratio': 1.65,          # 原始突破日量比
        'secondary_rules': [],               # 同天触发的其他规则（去重用，当前为空）
    }
}
```

### 5.2 关键字段说明

| 字段 | 必须 | 说明 |
|---|---|---|
| `date` | ✅ | 失败触发的交易日，framework 自动注入 `source` 字段 |
| `type` | ✅ | 固定 `'bearish'`，表示卖出信号 |
| `breakout_date` | ✅ | 指向原始突破信号，同名多信号归组 |
| `buy_point` | ✅ | 便于前端 tooltip 时间线直接展示，无需反查 |
| `breakout_low` | ✅ | 规则 B 阈值，也用于前端展示 |
| `failure_day` | ✅ | 突破后第几天触发，评估失败速度 |
| `failed_rule` | ✅ | 规则代号，前端映射颜色和图标 |
| `severity` | ✅ | 三级严重级别，决定五角星颜色 |
| `details` | ✅ | 扩展信息，前端 tooltip 和时间线详细展示 |

## 6. API 设计

### 6.1 回测端点

```
GET /api/breakout-failure
```

**参数**：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `code` | string | 必填 | 股票/指数代码 |
| `start` | string | 必填 | 起始日期 YYYY-MM-DD |
| `end` | string | 必填 | 截止日期 YYYY-MM-DD |
| `mode` | string | `stock` | `stock` / `index` |
| `monitor_days` | int | 10 | 覆盖 YAML 默认值 |
| `rule_*_enabled` | bool | true | 覆盖 YAML 默认值 |
| `rule_*_*` | float/int | YAML 默认 | 覆盖各规则参数 |

**行为**：API 内部静默将 `end` 延长 `min(monitor_days, 今天 - end)` 天查 K 线，确保监控窗口数据充足。返回 `klines_display_end` 指示前端渲染截止日。

**返回**：

```json
{
  "code": "600519",
  "name": "贵州茅台",
  "klines": [...],
  "klines_display_end": "2026-05-19",
  "breakout_signals": [...],
  "failure_signals": [...],
  "stats": {
    "total_breakouts": 15,
    "total_failures": 42,
    "failed_breakouts": 7,
    "failure_rate_pct": 46.7,
    "by_severity": { "mild": 12, "confirmed": 18, "severe": 12 }
  }
}
```

### 6.2 诊断端点

```
GET /api/breakout-failure/diag
```

**参数**：

| 参数 | 类型 | 说明 |
|---|---|---|
| `code` | string | 股票代码 |
| `breakout_date` | string | 要排查的原始突破日 YYYY-MM-DD |
| `check_date` | string | 要检查的日期 YYYY-MM-DD |
| `mode` | string | `stock` / `index` |

**返回**：

```json
{
```

- **检查日**日期选择器

### 7.5 配置持久化

符合回测看板铁律：`💾 保存配置` → POST YAML 到 `/api/config?signal_type=breakout_failure`，页面加载时 GET 回填。

### 7.6 K 线图集成

使用 `kline-chart.js`，突破信号和失败信号合并传入 `signals` 数组。每条信号可覆盖 `symbol`（pin / 五角星 SVG path）、`color`、`symbolSize`。

**五角星 SVG path**：
```
path://M16,0 L20,12 L32,12 L22,20 L26,32 L16,24 L6,32 L10,20 L0,12 L12,12 Z
```

**三级颜色**：
- mild → `#FF9800`（橙色）
- confirmed → `#9C27B0`（紫色）
- severe → `#000000`（黑色）

**堆叠偏移**：`y = close - range * 0.75 * (i + 1)`，range = high - low，i 为同天信号序号。

**kline-chart.js 改造**：`markPoint` 的 `symbol`、`symbolSize`、`itemStyle.color` 从每个数据点独立读取，全局值为兜底。约 10 行改动，API 签名不变。
