# 知行系统 — 产品需求书 (PRD)

> 版本：v1.0
> 日期：2026-05-20
> 状态：设计完成，待开发

---

## Problem Statement

独立 A 股投资者面临系统性约束缺失——没有人商量、没有人监督、没有人复盘、没有绩效归因。这导致：

- 冲动交易：看到一根大阳线就追入，没有人在旁边喊停
- 纪律松弛：制定了止损规则但临场自己改掉，小亏变大亏
- 情绪驱动：FOMO 怕错过、恐慌抛售，事后不知道为什么会亏钱
- 无法成长：赚了不知道是运气还是能力，亏了不知道是选股错还是大盘差

用户已有一套成熟的**选股和择时引擎**（CAN SLIM 七维评分、RS 强度排名、十余种形态扫描、见顶信号识别、大盘环境评估），但缺少一个将这些"知"转化为"行"的纪律执行层。

---

## Solution

「知行」系统——一个 AI 投资纪律官，补齐机构交易员天然具备但独立投资者缺失的四个约束角色：

- **风控总监**：硬拦截。超限单子根本发不出去
- **投资委员会**：软警告。审视每笔买入的理由和大盘时机
- **合规部门**：检查用户是否遵守自己设定的策略边界
- **绩效分析师**：把每笔盈亏拆成"市场给的"和"自己作的"，追踪情绪与决策的关联

系统不替用户决策，而是**在用户冲动时喊停，在复盘时让用户看清自己**。三阶段闭环：事前拦截 → 事中监控 → 事后复盘。

## User Stories

### 观察池浏览者（每日选股）

1. 作为一个投资者，我希望系统每天自动运行 RS 强度筛选，把满足条件的稳健龙头、短期爆发和双强股挑出来，放入观察池，这样我不用在 5000+ 只股票里大海捞针。
2. 作为一个投资者，我希望观察池中的每只股票都自动附带多维度扫描结果——CANSLIM 评分、财务健康、估值仪表、形态信号——这样我一眼就能判断这只股票值不值得深入看。
3. 作为一个投资者，我希望观察池每日全量更新，今天的观察池不残留昨天的过时标的，这样我始终面对最新的市场数据。
4. 作为一个投资者，我希望观察池按综合评分排序，高分股票排在前面，我可以快速聚焦最有潜力的标的。

### 股票复核者（深度研判）

5. 作为一个投资者，我希望点击观察池中任意一只股票，进入一个统一的复核详情页，在一个 K 线图上看到该股历史上所有引擎检测出的买入信号和卖出信号，用不同颜色区分，这样我可以直观判断是否出现择时买点。
6. 作为一个投资者，我希望在复核详情页上同时看到该股的买入信号汇总和风险提示，如果该股近期出现过见顶信号（高潮顶、铁轨线、量价背离），系统要明确警示我谨慎。
7. 作为一个投资者，我希望复核详情页下方展示 CANSLIM 七维评分卡片、RS 强度趋势、财务健康仪表、估值仪表——所有信息在一页内完成，不需要在多个页面之间跳来跳去。
8. 作为一个投资者，我希望复核页加载速度快——后端一次性返回该股全历史 K 线和所有引擎信号，而不是前端逐个调 API 聚合。

### 自选池管理者

9. 作为一个投资者，我希望在复核详情页上一键将股票从观察池确认加入自选池，这样我就可以在自选池中持续跟踪它。
10. 作为一个投资者，我希望可以手动添加任何股票到自选池——即使它没有被观察池自动筛选出来——但系统必须标注它为"手动添加"，并强制对其运行完整评估流程，在评估完成前禁止买入。
11. 作为一个投资者，我希望手动添加时系统弹窗问我"该股未通过观察池自动筛选，你的添加理由是什么？"，这个回答被记录下来，方便事后复盘。
12. 作为一个投资者，我希望可以从自选池中移除不再关注的股票。
13. 作为一个投资者，我希望自选池列表显示每只股票的评估状态（已评估/评估中/待评估），以及它是从观察池来的还是手动添加的。

### 买入决策者（事前拦截）

14. 作为一个投资者，当我从自选池中选择某只股票点击"计划买入"时，系统弹出买入表单，我必须填写买入股数和成交价。
15. 作为一个投资者，我希望在填写买入信息的同时，系统实时执行买入前检查清单——自动读取最新的大盘环境、CANSLIM 评分、RS 强度排名——并把结果展示在表单旁边，让我看到哪些检查通过了、哪些有风险。
16. 作为一个投资者，我希望系统对硬拦截规则（单票仓位上限、总仓位上限、止损价必须设置）执行严格管控——如果这些不通过，买入按钮保持灰色不可点击。
17. 作为一个投资者，我希望系统对软警告规则（大盘环境、CANSLIM 评分不达标、RS 排名不足、行业集中度过高）弹出明确警告，但我可以在勾选"已知风险"后继续操作——这样既保持了纪律又不能过度僵化。
18. 作为一个投资者，我希望在确认买入前必须填写止损价、目标周期、买入理由（至少 20 字），否则无法提交——写下 = 承诺。
19. 作为一个投资者，我希望在确认买入前看到后果预览——"如果你在止损价卖出，将亏损 X 元"——让我对最坏情况有心理准备。
20. 作为一个投资者，如果我分批买入同一只股票（今天买 500 股、下周加仓 300 股），每批都应该作为独立交易记录管理——各自有独立的买入价、止损价、买入理由——这样复盘时每笔决策的责任清晰可追溯。

### 持仓监控者（事中监控）

21. 作为一个投资者，我希望系统持续监控我所有持仓标的，每天自动检查：大盘环境是否恶化、RS 强度是否下滑、是否出现了见顶信号、是否接近止损线、是否触发 7 周法则。
22. 作为一个投资者，我希望持仓监控页面用红色/黄色/绿色直观显示每只股票的告警级别，有告警的标的排在前面。
23. 作为一个投资者，我希望每条告警都标注来源引擎——"大盘环境恶化（来源：market_direction_daily）"——让我知道这不是凭空判断。
24. 作为一个投资者，我希望系统综合所有引擎的输出，给出最终建议：持有 / 减仓 / 清仓，并列出理由摘要。

### 卖出记录与复盘者（事后复盘）

25. 作为一个投资者，当我要卖出一批持仓时，我可以选择对应的买入批次、填写卖出价格、卖出理由和卖出时的情绪状态，系统自动计算该批次盈亏金额、盈亏比例和持股天数。
26. 作为一个投资者，我希望复盘页面帮我做绩效归因——这笔盈利中有多少是大盘上涨带的（β 收益），多少是我的选股能力贡献的（α 收益）。
27. 作为一个投资者，我希望系统追踪我的规则遵守率——我买入时检查清单通过率多少？我是否在止损触发时执行了卖出？我是否在见顶信号出现时选择了忽视？
28. 作为一个投资者，我希望每次交易都标记买入和卖出时的情绪（冷静/兴奋/FOMO/焦虑/恐慌/沮丧/解脱），长期追踪让我发现"什么时候我最容易犯错"。
29. 作为一个投资者，我希望每次交易完成后给出一个心力评分（-2 到 +2），系统绘制成心力曲线，叠加到大盘 K 线图上，让我看到情绪周期与市场周期的关系。

### 年度回顾者（长期成长）

30. 作为一个投资者，我希望每年底系统生成一份"年度体检报告"——包含我的胜率、盈亏比、最大回撤、最佳和最差决策、规则遵守率趋势、心力曲线。
31. 作为一个投资者，我希望看到"规则回溯"结果——用历史数据重跑一遍，如果我 100% 遵守了所有规则，我的收益率曲线会是什么样子——让我直观感受纪律的货币价值。
32. 作为一个投资者，我希望看到"我的投资人格"雷达图，从选股能力、择时能力、纪律性、情绪控制、风控意识五个维度画像，了解自己作为投资者的优势和弱点。

## Implementation Decisions

### 架构决策

1. **规则分级机制**：检查清单规则按拦截力度分三级——硬拦截（触发则不可交易）、软警告（可勾选"已知风险"后继续）、合规检查（表单完整性，不满足不可提交）。
2. **API 硬拦截错误码**：买入 API 在硬拦截规则未通过时返回 HTTP 422 + 具体原因，前端必须显式处理而非静默忽略。
3. **规则热加载**：借鉴 VnPy 的 RuleTemplate 模式——每条规则独立一个模块，系统自动发现并加载。新增规则不改主代码。
4. **API Blueprint 模式**：交易和纪律相关 API 使用 Flask Blueprint 注册，不修改已有的 server.py 路由结构。

### 数据架构决策

5. **观察池每日全量替换**：观察池是日更快照表，每天 truncate + insert 新一批结果。用户的复核/确认操作状态独立存储在另一张状态表，不受每日替换影响。
6. **宽表策略**：观察池快照包含所有扫描维度的计算结果（RS 分类、CANSLIM 七维评分、财务健康、估值仪表、买入信号、卖出信号、综合建议），前端一次查询完成页面渲染，不跨表 JOIN。
7. **分批独立管理**：每批买入 = 一条独立的交易记录，盈亏按批次独立计算，不采用 FIFO 聚合。卖出时指定对应批次。
8. **已有数据表处理**：`portfolio_holdings` 迁移到新交易表后废弃；`portfolio_signals` 保留作 V2 告警验证参考；`buy_signals_daily` 废弃；`watchlist` 保留并扩展字段（来源、评估状态）。
9. **RS 分类规则**（排他性，避免重叠歧义）：

| 分类 | 条件 | 含义 |
|------|------|------|
| 稳健龙头 | RPS250≥90 且 RPS20≥85 | 长期极强 + 近期也强 |
| 加速爆发 | RPS250≥80 且 RPS20≥95 | 长期有底子 + 近期极强 |
| 短期爆发 | RPS20≥95 且不满足加速爆发 | 近期极强但长期不够 |
| 长期稳健 | RPS250≥85 且不满足稳健龙头 | 长期强但近期温和 |

双强股：同时满足稳健龙头和加速爆发（RPS250≥90 且 RPS20≥95），同时标两种标签。

### API 契约

9. **观察池**：`GET /api/discipline/observation` 返回今日全量快照，按综合评分倒序。
10. **复核详情**：`GET /api/discipline/review/<stock_code>` 一次性返回全历史 K 线 + 所有引擎信号（买入信号和卖出信号分组），前端不调多个 API。
11. **自选池**：`POST /api/discipline/watchlist` 确认加入；`POST /api/discipline/watchlist/manual` 手动添加（触发异步评估）；`GET /api/discipline/watchlist` 列表；`DELETE` 移除。
12. **买入检查**：`POST /api/discipline/precheck` 纯检查，不下单，返回每条规则的通过/警告/拒绝状态。
13. **买入下单**：`POST /api/discipline/trades` 包含完整检查清单结果，硬拦截未通过返回 422。
14. **卖出**：`PUT /api/discipline/trades/<id>` 填写卖出信息，自动计算盈亏、持股天数。
15. **盈亏汇总**：`GET /api/discipline/summary` 支持日期筛选。

### 模块划分

16. **观察池日更模块**：每日定时任务，执行 RS 分类筛选 + 多维度扫描，结果写入观察池宽表。扫描只对进入观察池的股票执行，不对全网计算。
17. **复核详情模块**：后端聚合所有已注册引擎，对单只股票做全历史信号扫描，输出统一的 K 线 + 信号时间序列 + 信息卡片数据。
18. **规则引擎模块**：包含 8 条检查规则（大盘环境、CANSLIM、RS、单票仓位、总仓位、行业集中度、止损、理由），每条规则独立可配置，引擎自动发现并串行执行。
19. **交易记录模块**：管理买入→卖出完整生命周期，包含检查清单快照、情绪标记、盈亏计算。
20. **持仓监控模块**（V2）：聚合已有引擎信号，对每笔持仓生成告警和综合建议。
21. **绩效归因模块**（V3）：α/β 分解、规则遵守率、情绪关联分析、规则回溯、心力曲线。

### 前端决策

22. **统一复核详情页**：一只股票 = 一个页面 = 一个 K 线图（复用已有通用 K 线组件）+ 所有引擎信号标记（颜色分层：绿=买入信号，红/橙=卖出信号）+ 信息卡片（左侧买入信号汇总，右侧风险提示）。
23. **买入表单内嵌检查清单**：检查清单结果实时展示在表单旁边，硬拦截未通过时买入按钮灰色禁用。

## Testing Decisions

### 测试策略

好测试的标准：只测试外部行为，不测试实现细节。测试应模拟用户的真实操作流程，验证端到端的行为正确性。

### 测试重点模块

- **规则引擎**：每条规则独立可测。输入 stock_code + 价格 + 数量 + 总资金，验证返回的 pass/warn/fail 状态是否符合预期。边界测试：仓位恰好等于上限、RS 排名恰好在前 20% 边界。
- **交易 CRUD API**：测试买入创建、买入检查拦截（硬/软/合规三级）、卖出更新、盈亏计算正确性。
- **观察池日更**：验证每日替换逻辑正确（旧数据清除、新数据插入），验证扫描只对观察池股票执行（不对全网）。
- **复核 API**：验证返回数据包含所有已注册引擎的信号，验证买入/卖出信号分组正确。

### 已有测试参考

项目中已有回测看板的开发标准（`docs/dev/回测看板开发标准.md`），新模块的测试风格应与此保持一致。

## Out of Scope

以下内容不在本 PRD 范围内：

- **实盘交易执行**：系统不做自动下单，不连接券商接口。用户手动填写成交价。
- **实时行情推送**：系统依赖盘后日 K 线数据，不做盘中实时行情。
- **加密货币 / 期货 / 期权**：仅聚焦 A 股。
- **多用户支持**：单用户系统。
- **移动端**：仅桌面 Web 前端。
- **AI 策略推荐**：系统根据规则给出建议（持有/减仓/清仓），但不训练模型代替用户做最终决策。
- **V2/V3 的全部功能**：V1 聚焦观察池 + 复核 + 自选池 + 检查清单 + 交易记录 + 盈亏概览。持仓监控（V2）和绩效归因（V3）在后续 PRD 中展开。

## Further Notes

- **命名**："知行"取自"知行合一"——知（已有扫描引擎）→ 行（纪律执行）。英文：O'Neil Discipline Engine。
- **设计哲学**：系统是"纪律官"而非"决策替身"。它在用户冲动时喊停，在复盘时让用户看清自己，但最终买卖决策权始终在用户手中。
- **与已有系统的关系**：知行系统作为消费者读取已有扫描引擎的输出，不修改引擎本身。所有已有引擎保持只读。
- **心理学基础**：检查清单借鉴 Atul Gawande 的《清单革命》（Checklist Manifesto）——强制慢思考覆盖快直觉。心力追踪借鉴行为金融学——将情绪量化才能管理情绪。
- **V1 预估工期**：6-8 天。
